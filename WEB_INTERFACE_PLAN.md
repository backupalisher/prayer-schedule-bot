# План реализации веб-интерфейса для системы мониторинга бота

## Архитектура

### Технологический стек
- **Бэкенд**: Flask (легковесный, простой в интеграции)
- **Фронтенд**: HTML/CSS/JavaScript с Bootstrap 5
- **База данных**: SQLite (существующая) + новая таблица для истории и планирования
- **Асинхронность**: Flask с поддержкой async/await

### Структура проекта
```
prayer_bot_web/
├── app.py                    # Основное Flask приложение
├── requirements_web.txt      # Дополнительные зависимости
├── static/
│   ├── css/
│   │   └── style.css        # Стили
│   ├── js/
│   │   └── dashboard.js     # JavaScript для dashboard
│   └── images/
│       └── holidays/        # Изображения для праздников
├── templates/
│   ├── base.html            # Базовый шаблон
│   ├── dashboard.html       # Главная страница
│   ├── history.html         # История сбоев
│   ├── settings.html        # Настройки
│   ├── schedule.html        # Планирование сообщений
│   └── messages.html        # Управление шаблонами сообщений
└── web_services/
    ├── __init__.py
    ├── models.py            # Модели данных
    ├── routes.py            # Маршруты Flask
    ├── database.py          # Расширение базы данных
    └── scheduler.py         # Планировщик для веб-интерфейса
```

## Функциональные модули

### 1. Dashboard для мониторинга состояния
- **Текущий статус бота**: онлайн/офлайн, последняя проверка
- **Статистика**: успешные/неуспешные отправки, перезапуски
- **Графики**: история доступности за последние 24 часа/7 дней
- **Быстрые действия**: ручной перезапуск, тестовая отправка

### 2. История сбоев и перезапусков
- **Таблица сбоев**: время, тип ошибки, длительность
- **Детали перезапусков**: причина, результат, время восстановления
- **Фильтры**: по дате, типу ошибки, статусу
- **Экспорт**: CSV/JSON для анализа

### 3. Управление настройками мониторинга
- **Интервал проверки**: настройка частоты мониторинга
- **Пороги сбоев**: настройка лимитов для автоматического перезапуска
- **Оповещения**: включение/выключение, настройка получателей
- **Интеграции**: настройка webhook, email, SMS

### 4. Планирование сообщений
- **Форма создания**: текст, дата/время отправки, получатели
- **Шаблоны сообщений**: предопределенные тексты для праздников
- **Прикрепление изображений**: выбор из загруженных изображений
- **Повторение**: разовое, ежедневное, еженедельное, ежемесячное
- **Управление очередью**: просмотр, редактирование, отмена запланированных

### 5. Управление изображениями
- **Загрузка изображений**: для праздников (Рамадан, Курбан-байрам и др.)
- **Галерея**: просмотр всех доступных изображений
- **Привязка к шаблонам**: выбор изображения для конкретного шаблона
- **Оптимизация**: автоматическое сжатие и конвертация

## Интеграция с существующей системой

### Модуль мониторинга
- Расширение класса `BotMonitor` для записи истории в БД
- Добавление методов для управления через веб-интерфейс
- Webhook для实时 обновления статуса на dashboard

### Планировщик
- Интеграция с существующим `APScheduler`
- Добавление задач из веб-интерфейса
- Синхронизация состояния между веб-интерфейсом и ботом

### База данных
- Новая таблица `monitor_history` для истории сбоев
- Новая таблица `scheduled_messages` для запланированных сообщений
- Новая таблица `message_templates` для шаблонов
- Новая таблица `uploaded_images` для изображений

## API Endpoints

### Мониторинг
- `GET /api/status` - текущий статус бота
- `GET /api/history` - история сбоев
- `POST /api/restart` - ручной перезапуск
- `POST /api/test` - тестовая отправка

### Планирование сообщений
- `GET /api/scheduled` - список запланированных сообщений
- `POST /api/schedule` - создание нового запланированного сообщения
- `PUT /api/schedule/<id>` - обновление запланированного сообщения
- `DELETE /api/schedule/<id>` - удаление запланированного сообщения

### Шаблоны и изображения
- `GET /api/templates` - список шаблонов
- `POST /api/templates` - создание шаблона
- `POST /api/upload` - загрузка изображения
- `GET /api/images` - список изображений

## Ограничения и considerations

### Производительность
- SQLite может стать узким местом при большой нагрузке
- Consider миграция на PostgreSQL при необходимости
- Кэширование часто запрашиваемых данных

### Масштабируемость
- Возможность запуска нескольких инстансов бота
- Балансировка нагрузки между инстансами
- Централизованное хранение состояния

### Безопасность
- Ограничение доступа к веб-интерфейсу
- Защита от DDoS атак
- Регулярное обновление зависимостей

## Заключение

Веб-интерфейс значительно расширит возможности управления ботом, предоставив:
1. Централизованный контроль над всеми аспектами работы
2. Удобный инструмент для планирования и управления сообщениями
3. Детальную аналитику и мониторинг состояния системы
4. Гибкую систему шаблонов для праздничных сообщений

Реализация займет примерно 10-15 дней разработки с учетом интеграции с существующей системой.
